# Controllers Instructions

These instructions apply to files inside `controllers/`.

## Goal

- Keep controllers thin, readable, and focused on request handling.
- Move repeated or complex logic into reusable helpers, utilities, or model methods.

## Controller Rules

- Avoid putting large chunks of business logic directly inside one controller file when it can be split cleanly.
- Extract shared behavior into `utils/`, model methods, or small helper modules when multiple actions need it.
- Keep handlers short by delegating validation, transformation, and formatting work to smaller functions.
- Prefer composing logic from reusable pieces over adding more branching inside a single action.
- When you split a controller, prefer one workflow or responsibility per file, keep helper names behavior-focused, and avoid scattering the same request path across too many tiny files.
- Keep the root controller file as the request entrypoint and place helper controller modules in a feature subdirectory such as `controllers/lead/` so the main directory stays easy to scan.
- When moving related controller helpers, keep the directory structure aligned with the feature instead of leaving mixed root-level helper files beside the entrypoint.

## Structure Rules

- Group related handlers together, but do not let one controller become a dumping ground for unrelated actions.
- If a controller file becomes hard to scan, split it by responsibility into smaller modules.

## Validation

- Check that controller refactors preserve request/response behavior.
- Run the smallest useful test or manual verification for the route touched by the change.
