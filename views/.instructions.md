# Views Instructions

These instructions apply to files inside `views/`.

## Goal

- Keep all pages visually consistent with the current CRM theme.
- Prefer shared partials and reusable fragments over page-specific duplication.

## EJS and UI Rules

- Reuse the existing shared partials under `views/partials/ui/` whenever possible.
- Keep page headers, flash messages, cards, tables, forms, and modals aligned with the established spacing and typography.
- Avoid introducing new layout patterns unless the task explicitly calls for it.
- If a change affects more than one page, update the shared partial instead of copying markup.
- Preserve mobile behavior and desktop layout parity.
- If a view gets large, break it into smaller partials or includes instead of keeping all markup in one file.
- When splitting views, organize partials by visible purpose or user flow, keep includes shallow, and make the parent EJS file read like a simple page outline.
- Keep the main EJS file focused on composition and data wiring, not repeated UI blocks.

## Styling Rules

- Use the existing CSS and Tailwind pipeline.
- Keep class names and component structure close to the surrounding code.
- Avoid one-off visual tweaks in individual views when a shared class or partial can solve it.

## Validation

- Check the impacted view on both mobile and desktop when the change is visual.
- Update the visual regression checklist if the page structure or reusable UI patterns change.
